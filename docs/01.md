## Decision: Coordinate Order Convention

**Context:**
Geospatial data uses coordinates, but there are two conventions:

- Geographic: [latitude, longitude] (N/S first, then E/W)
- GeoJSON/Mapbox: [longitude, latitude] (x-axis first, then y-axis)

**Decision:** Follow GeoJSON/Mapbox standard: **[longitude, latitude]**

**Rationale:**

1. Matches GeoJSON specification (RFC 7946)
2. Mapbox GL JS expects this order
3. Aligns with mathematical [x, y] convention
4. Industry standard for web mapping

**Implementation:**

```typescript
// Type definition makes it explicit:
coordinates: [number, number]; // [longitude, latitude]

// Amsterdam example:
coordinates: [4.9041, 52.3676];
//            └─ lng   └─ lat

// Helper function for display:
export const formatCoordinates = (coords: [number, number]): string => {
  const [lng, lat] = coords; // destructure in correct order
  return `${Math.abs(lat).toFixed(4)}°${lat >= 0 ? "N" : "S"}, 
          ${Math.abs(lng).toFixed(4)}°${lng >= 0 ? "E" : "W"}`;
};
```

**Common gotcha:**
When reading coordinates aloud or in documentation, humans expect "Amsterdam is at 52°N, 4°E" but our array is `[4, 52]`.

**Mitigation:**

- Always destructure as `[lng, lat]` to make intent clear
- Add TypeScript comments: `[number, number] // [longitude, latitude]`
- Use named properties where possible: `{ longitude, latitude }`
